# 02. Hardware

Three tiers. Pick based on what you have. All tiers assume Ubuntu 22.04 / 24.04, CUDA 12.4 (**not** 13.2 — breaks Gemma 4 GGUFs), driver ≥ 550, Python 3.11.

## Tier 1 — MVP (1× RTX 3090, 24GB)

**Model:** Gemma 4 E4B at 8-bit (~5GB) or Q4 (~3GB) GGUF.

**Topology:**
- One vLLM server on the single GPU, dual-role (proposer + solver via role tokens).
- 4 LoRA branches, rank 16, swapped between generations (only one loaded at a time).
- Python executor runs in Docker on CPU. No verifier farm.

**Throughput target:** ~15 problems/minute, ~1 generation per 20 minutes. 48-hour run = ~144 generations.

**What you validate:** The core AZR + Elo loop runs, anchor doesn't revert every generation, top branch stably beats bottom branch head-to-head.

**What you don't validate:** Multi-GPU pipeline, full-sized (26B) model capability, production sandbox, long-horizon stability beyond a weekend.

**Success criterion:** ≥5pp over base Gemma 4 E4B on held-out HumanEval+ after 48 hours.

## Tier 2 — Working scale (5× RTX 3090, 120GB total)

**Model:** Gemma 4 26B-A4B (MoE) at Q4 (~16GB) or 8-bit (~28GB, sharded).

**Topology:**
- **GPU 0:** Primary vLLM inference server. Gemma 4 26B Q4 loaded, all rollouts route here. Target throughput 300–600 tok/s via vLLM batched decode.
- **GPUs 1–3:** LoRA training workers. 4 LoRA branches per GPU (rank 32), swapped in/out. verl FSDP2 + GRPO. LoRA adapters are ~300MB each; checkpointing is cheap.
- **GPU 4:** **Verifier farm head.** Hosts Firecracker microVMs (CPU-bound work, GPU idle most of the time). Verifier throughput is the bottleneck at this tier. Also runs background SAE / analysis jobs if In-Place TTT is active.

**Population:** 12 solver branches, 4 proposer variants (temperature/prompt variations), 1 anchor-eval branch never updated.

**Throughput target:** ~80 problems/minute, 1 generation per ~8 minutes.

**Where real results emerge.** This is the tier where compounding should become visible across generations. Expect to spend ~2 weeks here per experimental iteration.

## Tier 3 — Exploration (10× RTX 3090, 240GB total)

**Model:** Gemma 4 31B Dense at 8-bit, sharded across GPUs 0–1 via tensor parallelism. Higher fidelity generation, especially for induction tasks.

**Topology:**
- **GPUs 0–1:** Gemma 4 31B 8-bit, tensor-parallel. vLLM with `--tensor-parallel-size 2`.
- **GPUs 2–7:** 6× parallel LoRA branch workers, 4 branches each. 24 total solver branches.
- **GPU 8:** In-Place TTT training head (once Phase 2 is active). Also runs per-branch meta-learning if we add it.
- **GPU 9:** Verifier farm + live anchor eval + analysis.

**Population:** 24 solvers, 8 proposers, 2 anchor branches (one for HumanEval+, one for LiveCodeBench held-out).

**Throughput target:** ~250 problems/minute, 1 generation per ~3 minutes.

**What to actually try here that you can't try lower:** multi-domain curricula (code + math + formal reasoning), longer Elo tournaments with higher K (more population turnover), In-Place TTT experiments in the hot loop.

## Non-GPU requirements

- **RAM:** ≥128GB system RAM at Tier 2+. verl with FSDP2 CPU offload uses host RAM aggressively; too little and training hangs or OOMs without clear errors.
- **Storage:** ≥2TB NVMe. Checkpoints per generation (every LoRA branch) + rollout traces + anchor eval results + W&B cache add up fast. Budget ~50GB/day of accumulated state at Tier 2.
- **Network:** Irrelevant — everything runs local. But pre-download all weights, datasets, and pinned dependencies to a single directory (`$SI_CACHE`) before starting the run. Network failures mid-run break reproducibility.

## The verifier-farm rule

**The single most important piece of hardware advice in this repo:** at Tier 2+, verifier latency is the bottleneck, not inference throughput. Every problem generated by the proposer needs to be executed — and often multiple times if Monte Carlo proposer reward (AZR §3.3) requires K=8 rollouts to estimate solver success probability. That's 8+ Python executions per proposal. At 80 proposals/minute, you're running 640+ executions/minute.

If your verifier runner averages 500ms cold-start, you need at minimum 5 parallel runners to keep up. Scale linearly with proposal rate.

**Do not buy the next GPU before you've saturated verifier throughput.** Migrating from `subprocess.run(..., timeout=5)` to Firecracker (~50ms cold, <1ms hot for a pooled pool) is typically a 10× throughput win, comparable to adding a GPU, at zero GPU cost.

See [`04-implementation.md`](04-implementation.md) §3 for the concrete verifier setup.

## What you don't need

- **H100s, A100s, or cloud.** The premise is home-scale. If you have access to bigger hardware, the stack scales but the design doesn't change — you're just buying throughput, not capability.
- **Custom PSU / liquid cooling for 3090s.** 3× 3090s on a 1000W PSU is stable. 5× needs a 1500W. 10× needs dual-PSU or server chassis. Power draw peaks during LoRA backward passes, not inference.
- **InfiniBand or NVLink.** Not required at these tiers. PCIe 4.0 x16 per GPU is sufficient when LoRA branches don't gradient-sync across boundaries (they don't, by design — each branch is independent).
